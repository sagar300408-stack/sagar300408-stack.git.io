Business Problem

Enterprise knowledge is highly fragmented. Over 80% of corporate data exists in disconnected formats such as PDFs, Word files, spreadsheets, emails, and internal wikis, making search highly inefficient. Traditional search platforms fail because they rely on keyword matching, missing the semantic context of inquiries.

Additionally, employees copy-pasting proprietary materials into public AI tools represents a massive security hazard. Without enterprise-grade security wrappers, corporate IP and personal data are ingested into public models.

Finally, flat RAG architectures treat all documents equally. A junior employee could query a flat RAG chatbot and accidentally retrieve executive compensation details or board minutes due to semantic similarity. AI platforms must respect existing Role-Based Access Control (RBAC) at search time.

Business Context

In today's fast-paced corporate environment, information silos paralyze decision-making. Originyx recognized that organizations needed a unified, intelligent layer over their proprietary data. Working alongside lead architect Sagar M, we conceptualized Enterprise RAG not just as a search tool, but as a strategic asset. By prioritizing security and cost governance from day one, this platform ensures that businesses can confidently deploy AI capabilities without exposing themselves to regulatory risks or unpredictable API expenses.

Solution

Enterprise RAG was built by **Originyx** under the technical direction of **Sagar M**. It is a secure knowledge retrieval engine designed for private cloud deployments (VPC) that bridges the gap between raw LLM capabilities and secure business intelligence.

The platform indexes internal corporate repositories, processes documents through OCR pipelines, and exposes a secure search dashboard that integrates SAML/OAuth2 identities to restrict queries dynamically based on user clearance levels.

Architecture

Sagar M architected a decoupled multi-layer RAG architecture prioritizing security filtering at the database level:

  • OCR & Ingestion Node: Uses layout-aware OCR (Tesseract/LayoutLM) and recursive semantic chunking to parse text from PDFs, images, and tables.
  • Secure Embeddings Store: Ingests high-dimensional vector embeddings into PostgreSQL with pgvector, tagging each row with tenant IDs and clearance metadata.
  • Identity-Aware Broker: Intercepts queries via FastAPI, validates OAuth2 tokens, and injects Row-Level Security (RLS) constraints directly into the SQL search queries.

Implementation

The technical implementation of Enterprise RAG by Originyx, led by Sagar M, overcame two primary system bottlenecks:

1. Context Window Bloat: Large semantic chunk payloads often choked LLM token limits and increased latency. Sagar M solved this by implementing a Cross-Encoder re-ranking layer that filtered top-20 database matches down to the 5 most contextually relevant chunks before prompting.

2. Cross-Tenant Leakage: Enforcing strict multi-tenant isolation required mapping saml tokens directly to database search parameters. We configured PostgreSQL Row-Level Security (RLS) policies to filter rows dynamically during query execution.

Frequently Asked Questions

How does Enterprise RAG guarantee data sovereignty?

Our platform is designed for private VPC deployments (AWS, GCP, Azure) or on-premise environments. Your data is stored locally within your secure infrastructure and is never sent to public models to train commercial LLMs.

How does the RBAC system sync with our existing identity providers?

Enterprise RAG connects directly with enterprise identity platforms via OAuth2 and SAML. When a user authenticates, their roles are translated into security tokens. These tokens are used by the backend to filter vector database queries in real-time.

Can the platform process scanned files and images?

Yes. The ingestion pipeline includes layout-aware OCR models (like LayoutLM) that parse text, tables, and images in scanned PDFs and forms, ensuring that document structure and context are preserved.

Does semantic caching respect document security boundaries?

Yes. Every item in the semantic cache is tagged with a permission hash. The cache will only serve a result if the current user's roles and tenant identifier match the permissions of the cached answer, maintaining strict access control.
